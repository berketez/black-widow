## 1. Problem Statement

Karadul v1.0/v1.1 can detect WHAT algorithms exist in individual functions but cannot trace HOW they work together across function boundaries. Three critical gaps:

### Gap 1: Obj-C Dynamic Dispatch (GlobalProtect case)

The GlobalProtect binary (10,888 functions, ~9,383 decompiled files) contains **12,860 `_objc_msgSend` calls**. These are virtual dispatches -- Ghidra's static call graph records edges to `_objc_msgSend` itself, not to the actual method implementations. Example from `GPStartVPNTunnel.c`:

```c
_objc_msgSend(lVar11, "startVPNTunnelWithOptions:andReturnError:", uVar5, &local_d8);
```

The actual `startVPNTunnelWithOptions:andReturnError:` method body exists somewhere in the binary as a separate function, but the call graph edge points to address `10028bef8` (the `_objc_msgSend` trampoline), not to the method implementation. This makes the call graph useless for tracing control flow through Obj-C code.

**Current state:** Ghidra's `ghidra_call_graph.json` has 32,733 edges, but a huge fraction terminate at `_objc_msgSend`/`_objc_msgSendSuper2` -- dead ends.

### Gap 2: Inter-Procedural Data Flow

In CalculiX-style FEA code, a stiffness matrix K is allocated in one function, populated element-by-element in another, assembled in a third, and finally used in Newton-Raphson solving in a fourth. Karadul detects "Gauss Quadrature" in function A and "LU Decomposition" in function D but has no concept of the data pipeline connecting them.

**Current state:** `confidence_calibrator.py` has rudimentary `EXPECTED_CALLEES` (e.g., Newton -> LU_solve) but this is a hardcoded adjacency check, not actual data flow analysis.

### Gap 3: Algorithm Composition

A VPN tunnel setup is a composition of DNS resolution, TLS handshake, authentication, tunnel setup, and packet encryption. These are separate algorithm detections that Karadul presents as a flat list. No grouping, no ordering, no "this calls that which feeds into the other" understanding.

**Current state:** The v1.1 `engineering_analysis.json` output lists algorithms per function. No cross-function composition view exists.

## 3. Module 1: Virtual Dispatch Resolver

### 3.1 Purpose

Parse `objc_msgSend(receiver, "selector:", ...)` calls from decompiled C code, resolve each selector to its actual method implementation using Ghidra's metadata, and produce an augmented call graph with "virtual edges" connecting dispatch sites to resolved targets.

Also handle C++ vtable dispatch and Swift witness_method (future extension, scaffolded).

### 3.2 Algorithm

**Phase 1: Collect dispatch sites**

Scan all decompiled C files for patterns:

```
Pattern 1 (Obj-C direct):
    _objc_msgSend(RECEIVER, "SELECTOR", ...)
    _objc_msgSendSuper2(RECEIVER, "SELECTOR", ...)

Pattern 2 (Obj-C variable selector):
    _objc_msgSend(RECEIVER, SEL_VAR, ...)
    where SEL_VAR was loaded from a selector reference

Pattern 3 (C++ vtable):
    (**(code **)(*(long *)RECEIVER + OFFSET))(RECEIVER, ...)
    i.e., indirect call through vtable pointer

Pattern 4 (Swift witness):
    swift_conformsToProtocol / witness_method patterns
```

For Obj-C, extract:
- Caller function name (from file name or function header)
- Caller address
- Selector string (the "method:name:" literal)
- Receiver expression (for class inference: `&_OBJC_CLASS___NSFoo` means class NSFoo)

**Phase 2: Build selector-to-implementation map**

From `ghidra_functions.json`:
- Functions with Obj-C naming convention: `methodName:` or `[ClassName methodName:]`
- The GlobalProtect data shows functions like `initWithWindowNibName:`, `loadWindow`, `startVPNTunnelWithOptions:andReturnError:`
- Parameters with `typedef SEL qword` or `typedef ID qword` identify Obj-C methods

From `ghidra_strings.json`:
- Selector strings and their addresses (cross-reference to `__objc_methnames` section)

From `ghidra_xrefs.json`:
- `function_xrefs[addr].strings_used` provides selector strings per function

From `ghidra_types.json`:
- `structures` section contains Obj-C class metadata (`_OBJC_CLASS___ClassName`)
- Can map class hierarchy for polymorphism resolution

**Phase 3: Resolve dispatch**

For each dispatch site `(caller, selector)`:

1. **Exact match:** Find a function named exactly `selector` in `ghidra_functions.json`
2. **Class-qualified match:** If receiver class is known (e.g., `&_OBJC_CLASS___NEVPNManager`), look for `[NEVPNManager selector]` or `+[NEVPNManager selector]`
3. **Fuzzy match:** If selector contains colons (`startVPNTunnelWithOptions:andReturnError:`), search function names containing all keyword parts
4. **External resolution:** If no match in binary, mark as "external_framework" (system framework method)

Each resolution gets a confidence score:
- Exact function name match: 0.95
- Class-qualified match: 0.90
- Partial keyword match with single candidate: 0.75
- Multiple candidates: 0.50 (ambiguous)
- Unresolved: 0.0 (marked external)

**Phase 4: Build augmented call graph**

For each resolved dispatch:
- Add virtual edge: `{from: caller_addr, to: resolved_addr, type: "objc_dispatch", selector: "...", confidence: X}`
- Keep original `_objc_msgSend` edges (don't remove them, mark as `type: "trampoline"`)

class DispatchSite:

OBJC_MSGSEND_RE = re.compile(

    r'(?:\s*,\s*([^)]*))?\)',            # optional remaining args

    r'(\w+)\s*'                          # selector variable
    r'(?:,\s*([^)]*))?\)',
)

# Receiver class extraction: &_OBJC_CLASS___ClassName
OBJC_CLASS_RE = re.compile(r'&_OBJC_CLASS___(\w+)')

# C++ vtable dispatch: (*(code **)(*(long *)obj + OFFSET))(obj, ...)
VTABLE_DISPATCH_RE = re.compile(
    r'\(\*\*?\s*\(code\s*\*+\)\s*'
    r'\(\*\(long\s*\*\)\s*(\w+)\s*\+\s*(\w+)\)\)'

## 4. Module 2: Inter-Procedural Data Flow Tracker

### 4.1 Purpose

Track data objects (stiffness matrices, encryption keys, socket descriptors, etc.) as they flow across function boundaries through parameters, return values, and global variables. Build a data dependency graph showing which functions produce, transform, and consume each data object.

### 4.2 Algorithm

**Phase 1: Extract function signatures and call sites**

From `ghidra_functions.json`:
- Function name, address, parameter list (name, type, ordinal)
- Return type

From augmented call graph (Module 1 output):
- Caller-callee relationships with resolved virtual edges

**Phase 2: Intra-procedural argument tracking (lightweight)**

For each function body (decompiled C), track how arguments flow:

```
Rule 1: Direct pass-through
    void foo(int *K, int n) {
        bar(K, n);  // K flows from foo to bar as arg 0
    }
    => DataFlow(source=foo.param_0, sink=bar.param_0, type="passthrough")

Rule 2: Return value capture
    K = allocate_matrix(n);
    fill_matrix(K, data);
    => DataFlow(source=allocate_matrix.return, sink=fill_matrix.param_0, type="return_to_arg")

Rule 3: Global variable mediation
    global_K = compute_K();
    ...
    solve(global_K, f);
    => DataFlow(source=compute_K.return, sink=global_K, type="global_write")
       DataFlow(source=global_K, sink=solve.param_0, type="global_read")

Rule 4: Struct member flow
    ctx->K = allocate();
    ...
    solve(ctx->K, ctx->f);
    => DataFlow through ctx struct member
```

This is NOT a full abstract interpretation. It is pattern-matching on the decompiled C text -- sufficient for Ghidra output where variable names are consistent within a function.

**Phase 3: Build inter-procedural data flow graph**

Nodes: Functions
Edges: Data flow connections with labels

```
Edge: {
    from: "allocate_stiffness",
    to: "element_stiffness",
    data_name: "K" (or "param_1" if unresolved),
    data_type: "double *" (if recoverable),
    flow_type: "return_to_arg" | "passthrough" | "global" | "struct_member",
    confidence: 0.7,
}
```

**Phase 4: Identify data objects and their lifecycles**

Group edges by data identity (same variable/pointer flowing through multiple functions):

```
DataObject {
    name: "stiffness_matrix_K"   (or "param_1_flow_001" if unnamed)
    inferred_type: "double *"
    lifecycle: [
        {function: "allocate_stiffness", role: "allocator", order: 0},
        {function: "element_stiffness",  role: "writer",    order: 1},
        {function: "assemble_global",    role: "modifier",  order: 2},
        {function: "newton_solve",       role: "consumer",  order: 3},
        {function: "free_matrix",        role: "deallocator", order: 4},
    ]
}
```

GLOBAL_ASSIGN_RE = re.compile(

### 4.5 Pointer Aliasing (Basic)

Full pointer alias analysis is a PhD-level problem. We use these pragmatic heuristics:

1. **Same-name tracking:** If function A passes variable `K` to function B's `param_1`, and function B passes `param_1` to function C, we infer K flows A -> B -> C.

2. **Register/stack slot continuity:** Within a single function, if `lVar5 = foo(...)` and later `bar(lVar5, ...)`, we track `lVar5` as the same data object.

3. **Type-based grouping:** If multiple flows carry the same type (e.g., `double *`) through related functions (same domain, adjacent in call graph), they likely represent the same logical object.

4. **No pointer arithmetic tracking:** We do NOT try to determine if `*(ptr+8)` and `*(ptr+16)` alias. Too unreliable on decompiled code.

## 5. Module 3: Algorithm Composition Analyzer

### 5.1 Purpose

Combine the augmented call graph, data flow graph, and algorithm detections to discover "algorithm compositions" -- groups of functions that together implement a higher-level protocol or workflow. Generate human-readable composition descriptions.

### 5.2 Algorithm

**Phase 1: Build algorithm-annotated graph**

Overlay algorithm detections onto the augmented call graph:

```
For each function in call graph:
    function.algorithms = [list of AlgorithmMatch for this function]
    function.domain = DomainClassification for this function
```

**Phase 2: Detect composition patterns**

Five composition patterns, detected by graph analysis:

```
Pattern 1: PIPELINE
    A -> B -> C -> D, each with an algorithm detection
    AND data flows A -> B -> C -> D (same data object)
    Example: allocate_K -> element_stiffness -> assemble -> solve

Pattern 2: PROTOCOL_SEQUENCE
    A calls B, C, D in sequence (detected from call order in decompiled code)
    AND each has different algorithm categories
    Example: dns_resolve -> tls_handshake -> authenticate -> tunnel_setup

Pattern 3: FORK_JOIN
    A calls B and C independently, then D uses results of both
    Example: compute_K and compute_f both feed into solve(K, f)

Pattern 4: ITERATIVE_REFINEMENT
    A calls B in a loop, with convergence check
    Example: newton_step() called repeatedly until convergence

Pattern 5: PRODUCER_CONSUMER
    A produces data, B consumes it (via return value, global, or parameter)
    Not necessarily adjacent in call graph
    Example: key_exchange() produces session key, encrypt_packet() uses it
```

**Phase 3: Build composition structures**

```
Composition {
    id: "comp_001",
    name: "VPN Tunnel Establishment",   // Generated from constituents
    pattern: "PROTOCOL_SEQUENCE",
    domain: "crypto",                    // Dominant domain
    functions: [
        {name: "dns_resolve",     order: 0, algorithm: "DNS Resolution",    role: "initiator"},
        {name: "tls_handshake",   order: 1, algorithm: "TLS 1.3 Handshake", role: "security_setup"},
        {name: "authenticate",    order: 2, algorithm: "SAML Auth",         role: "authentication"},
        {name: "setup_tunnel",    order: 3, algorithm: "IPSec Setup",       role: "tunnel_creation"},
        {name: "encrypt_packet",  order: 4, algorithm: "AES-256-GCM",      role: "data_protection"},
    ],
    data_flows: [...],                   // From Module 2
    call_chain: [...],                   // From augmented call graph
    confidence: 0.82,
    description: "VPN tunnel is established through a 5-step protocol: ...",
}
```

**Phase 4: Generate human-readable reports**

For each composition, generate:
1. A Mermaid flowchart
2. A prose description
3. Algorithm-annotated call tree (ASCII)

### 5.3 Composition Naming Heuristics

The composition name is derived from:

1. **Dominant algorithm names:** If the composition contains "TLS Handshake" + "AES Encryption" + "Certificate Validation", the name becomes "TLS-based Secure Communication Pipeline"

2. **Domain keywords:** From `DomainReport.primary_domain`:
   - crypto -> "Security Protocol"
   - structural -> "FEA Computation Pipeline"
   - fluid -> "CFD Solution Pipeline"

3. **Function name keywords:** Extract common substrings from participating function names:
   - `GPStartVPNTunnel`, `GPStopVPNTunnel`, `loadPreferencesForVPNMgt` -> "VPN Tunnel Management"

4. **Known protocol templates:** Hardcoded templates for common compositions:

### 5.5 Graph Clustering Algorithm

To discover compositions, we use **algorithm-weighted connected components** on a subgraph:

1. Build subgraph S from the augmented call graph:
   - Include only nodes that have at least one algorithm detection
   - Include edges between algorithm-bearing nodes (direct or 1-hop through non-algorithm nodes)
   - Weight edges by:
     - `w_call = 0.3` (call graph edge)
     - `w_data = 0.5` (data flow edge)
     - `w_domain = 0.2` (same domain bonus)

2. Find connected components in S where edge weight >= threshold (0.3)

3. For each component, classify the composition pattern by analyzing the subgraph structure:
   - Linear chain with data flow = PIPELINE
   - Linear chain without strong data flow = PROTOCOL_SEQUENCE
   - Fan-out then fan-in = FORK_JOIN
   - Cycle with convergence check = ITERATIVE_REFINEMENT
   - Single producer, multiple consumers = PRODUCER_CONSUMER

4. Rank compositions by total confidence (sum of step confidences * edge weights)

## 6. Module 4: Deep Call Chain Tracer

### 6.1 Purpose

Given a "target function" (e.g., `GPStartVPNTunnel`), trace ALL code paths up to a configurable depth, building an execution tree annotated with algorithm detections, data flows, and composition membership.

### 6.2 Algorithm

**Phase 1: Rooted BFS/DFS from target function**

Starting from the target function, explore the augmented call graph:

```
trace_tree(target, depth=10):
    root = TraceNode(target)
    queue = [(root, 0)]
    visited = set()

    while queue:
        node, level = queue.pop(0)
        if level >= max_depth or node.address in visited:
            continue
        visited.add(node.address)

        # Get callees from augmented graph (includes virtual dispatch)
        for callee in get_callees(node.address):
            child = TraceNode(callee)
            child.algorithms = lookup_algorithms(callee)
            child.data_flows = lookup_data_flows(node, callee)
            child.composition = lookup_composition(callee)
            node.children.append(child)
            queue.append((child, level + 1))
```

**Phase 2: Annotate nodes**

Each node in the trace tree gets:
- Algorithm detections (from v1.0/v1.1 results)
- Data flow annotations (from Module 2)
- Composition membership (from Module 3)
- Domain classification (from DomainClassifier)
- Call type: "direct", "objc_dispatch", "vtable", "external"
- Source location: file and line number

**Phase 3: Prune and rank**

The raw tree can be huge (10 levels * 10 callees/function = 10^10 nodes). Pruning strategies:

1. **Skip library functions:** Functions matching `_objc_retain`, `_objc_release`, `_objc_autorelease*`, `_dispatch_*`, `_NSLog`, `_CFRelease`, etc. are marked as "runtime" and not expanded further.

2. **Skip repeated subtrees:** If function F appears at depth 3 and depth 7 with the same callees, don't expand it twice.

3. **Algorithm priority:** Expand algorithm-bearing functions first. Non-algorithm leaf functions are pruned at depth > 5.

4. **Configurable filters:**
   - `include_patterns`: Only expand functions matching these patterns
   - `exclude_patterns`: Skip functions matching these patterns
   - `min_algorithm_depth`: Below this depth, only show algorithm-bearing functions

### 6.5 Target Selection Strategy

When no explicit target is specified, automatically select interesting entry points:

1. **Functions with high callee count:** Orchestrator functions (many callees = likely important)
2. **Functions with algorithm detections and callees:** "Root" of a computation tree
3. **Functions matching user-facing patterns:** `*Start*`, `*Init*`, `*Main*`, `*Run*`, `*Execute*`
4. **Composition root functions:** First function in each discovered composition

Default: trace the top-5 by `callee_count * (1 + n_algorithms)`.

### 11.1 GlobalProtect (10,888 functions, ~9,383 C files)

| Module | Operations | Est. Time | Memory |
|--------|-----------|-----------|--------|
| VirtualDispatchResolver | Scan 9383 files for dispatch patterns + resolve 12,860 sites | 8-15s | ~200MB (call graph + functions in memory) |
| InterProceduralDataFlow | Parse 9383 files for arg tracking + build flow graph | 15-25s | ~300MB (all function signatures + bodies cached) |
| CompositionAnalyzer | Graph clustering on ~500 algorithm-bearing nodes | 1-3s | ~50MB |
| DeepCallChainTracer | 5 traces * BFS to depth 10 | 2-5s | ~100MB |
| **Total v1.2 overhead** | | **26-48s** | **~400MB peak** |

### 11.2 CalculiX-scale (40,000 functions)

| Module | Operations | Est. Time | Memory |
|--------|-----------|-----------|--------|
| VirtualDispatchResolver | Scan 40K files (mostly no Obj-C dispatch, fast skip) | 5-10s | ~400MB |
| InterProceduralDataFlow | Parse 40K files for arg tracking | 40-60s | ~800MB |
| CompositionAnalyzer | Graph clustering on ~2000 algorithm-bearing nodes | 5-10s | ~100MB |
| DeepCallChainTracer | 5 traces * BFS to depth 10 | 3-8s | ~150MB |
| **Total v1.2 overhead** | | **53-88s** | **~800MB peak** |

### 11.3 Optimization strategies

1. **Parallel file scanning:** Both VirtualDispatchResolver and InterProceduralDataFlow scan decompiled files. They CAN be parallelized internally using `ThreadPoolExecutor(max_workers=CPU_PERF_CORES)` -- same pattern as `EngineeringAlgorithmAnalyzer._analyze_file`.

2. **Lazy file loading:** Don't read all 40K files into memory. Read one at a time, extract results, discard body.

3. **Early termination for dispatch resolver:** If binary has no `_objc_msgSend` in the call graph edges, skip the entire Obj-C resolution (check `ghidra_call_graph.json` edge targets for `_objc_msgSend`). This makes CalculiX essentially free.

4. **Shared file cache:** If both Module 1 and Module 2 need to scan decompiled files, do a single pass that extracts both dispatch sites and argument flows. This requires a `CombinedFileScanner` helper (optional optimization, not MVP).

## 15. Risk Assessment

### 15.1 HIGH RISK: Obj-C selector resolution accuracy

**Problem:** Multiple classes can implement the same selector. `connection` could be `[NEVPNManager connection]`, `[NSURLSession connection]`, etc.

**Mitigation:** Class-qualified resolution using receiver expression analysis. When receiver class is known (from `&_OBJC_CLASS___X` pattern), resolution is high-confidence. When unknown, mark as ambiguous with multiple candidates.

**Fallback:** Even unresolved dispatch sites are useful -- the selector string itself is valuable metadata that the current system completely ignores.

### 15.2 MEDIUM RISK: Data flow false positives

**Problem:** Pattern matching on decompiled code is inherently noisy. `bar(param_1)` might be a coincidence -- `param_1` in function A and `param_1` in function B are completely different variables.

**Mitigation:** Type-based filtering (only connect flows with compatible types), call graph confirmation (only connect flows where there's a call edge), confidence scoring (pattern match alone = 0.5, with type match = 0.7, with call edge = 0.85).

### 15.3 MEDIUM RISK: Composition over-clustering

**Problem:** Algorithm-weighted clustering might group unrelated functions into a single "composition" if they share a domain.

**Mitigation:** Require data flow evidence OR direct call chain between members. Domain alone is not sufficient -- there must be a structural (call graph or data flow) link.

### 15.4 LOW RISK: Performance on large binaries

**Problem:** 40K function binary with inter-procedural analysis.

**Mitigation:** The analysis is O(E) where E = edges, not O(N^2). With lazy file loading and parallel scanning (existing pattern from `EngineeringAlgorithmAnalyzer`), even 40K functions should complete in under 90 seconds.

### 15.5 LOW RISK: Memory usage

**Problem:** Augmented call graph for 40K functions could be large.

**Mitigation:** The original `ghidra_call_graph.json` for GlobalProtect (10K functions) is 12.9MB. Augmented version with virtual edges adds ~30%. For 40K functions, expect ~50MB in memory. Well within 800MB peak estimate.

## Appendix B: Compatibility Notes

1. **All v1.2 modules are optional.** If `enable_deep_trace: false` (or modules not importable), the pipeline runs exactly as v1.1.

2. **No changes to existing data structures.** `AlgorithmMatch`, `CAlgorithmResult`, `DomainReport` etc. are unchanged. New modules consume them as-is.

3. **Output format is additive.** Four new JSON files are added to `reconstructed/`. No existing file format changes.

4. **Thread safety.** New modules use the same `ThreadPoolExecutor(max_workers=CPU_PERF_CORES)` pattern for file scanning. No global mutable state.

5. **Memory.** New modules load Ghidra JSONs once, process, then release. The augmented call graph is the largest new in-memory structure (~15MB for 10K functions).
