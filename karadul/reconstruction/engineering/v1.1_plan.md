## 2. Module 1: Struct Recovery Engine

### 2.1 Purpose

The existing `CTypeRecoverer` (c_type_recoverer.py, ~550 LOC) already detects
`*(type *)(ptr + offset)` patterns and synthesizes struct layouts. However, it
operates purely syntactically -- it has no knowledge of what the struct
*represents*. Module 1 enriches recovered structs with algorithm context:

- If the function is identified as `newton_raphson` (by EngineeringAlgorithmAnalyzer),
  and a struct is accessed through `param_18`, the engine can infer that
  `param_18` is likely `ElementData` or `SolverContext`.
- Uses Ghidra's `ghidra_types.json` as seed data for known type info.
- Uses call graph to propagate struct identity across functions.

class StructRecoveryEngine:
    """Algorithm-aware struct recovery.

    Builds on CTypeRecoverer's output by:
    1. Matching recovered structs to known algorithm parameter signatures
    2. Naming fields using domain-specific templates
    3. Propagating struct identity through call graph
    4. Rewriting pointer+offset access to struct->field syntax
    """

    def __init__(self, config: Config) -> None: ...

    def recover(
        self,
        decompiled_dir: Path,
        functions_json: Path,
        ghidra_types_json: Path,
        call_graph_json: Path,
        output_dir: Path,
        *,
        existing_type_result: CTypeRecoveryResult | None = None,
        algorithm_matches: list[AlgorithmMatch] | None = None,
        domain_report: DomainReport | None = None,
    ) -> StructRecoveryResult: ...

    # --- Internal methods ---

    def _load_ghidra_types(self, path: Path) -> dict[str, Any]: ...
    def _load_call_graph(self, path: Path) -> dict[str, list[str]]: ...

    def _match_struct_to_algorithm(
        self,
        struct: RecoveredStruct,
        func_algorithms: dict[str, list[AlgorithmMatch]],
    ) -> tuple[str, str, float] | None:
        """Try to match a struct to a known algorithm parameter pattern.

        Returns (struct_name, algorithm_context, confidence) or None.
        Uses ALGORITHM_STRUCT_TEMPLATES below.
        """

    def _name_fields_from_template(
        self,
        struct: RecoveredStruct,
        template_name: str,
    ) -> list[EnrichedField]:
        """Apply domain-specific field naming template."""

    def _propagate_through_call_graph(
        self,
        enriched: dict[str, EnrichedStruct],
        call_graph: dict[str, list[str]],
        func_param_map: dict[str, dict[int, str]],
    ) -> dict[str, EnrichedStruct]:
        """If caller passes known struct to callee, propagate identity."""

    def _rewrite_access_patterns(
        self,
        c_code: str,
        struct_map: dict[str, EnrichedStruct],
        base_ptr_to_struct: dict[str, str],
    ) -> str:
        """Rewrite *(type*)(ptr + N) to struct->field_name."""

    def _emit_types_header(
        self,
        structs: list[EnrichedStruct],
        output_path: Path,
    ) -> Path:
        """Generate enriched types.h with comments and proper names."""

### 2.10 Key Algorithm Description

1. **Template Matching Phase:**
   - For each function with algorithm detections, look up `ALGORITHM_STRUCT_TEMPLATES`.
   - For each RecoveredStruct accessed in that function, compute a similarity score
     between the struct's field offsets/types and the template's expected layout.
   - Similarity = weighted sum of (offset_match * 0.4 + type_match * 0.3 + size_match * 0.3).
   - Accept matches above 0.5 confidence.

2. **Call Graph Propagation Phase:**
   - Build a map: function_name -> {param_index: struct_name}.
   - Walk call graph edges: if `caller` passes `known_struct` as arg `i` to `callee`,
     mark `callee.param_i` as the same struct.
   - Iterate until fixpoint (max 5 iterations, typically converges in 2).

3. **Rewrite Phase:**
   - For each C file, find all `*(type *)(base + offset)` patterns.
   - If `base` maps to a known struct and `offset` maps to a known field, rewrite to
     `base->field_name`.
   - Preserve original as inline comment: `base->field_name  /* was: *(int *)(base + 0x10) */`

## 3. Module 2: Semantic Parameter Naming

### 3.1 Purpose

The existing `CVariableNamer` (c_namer.py, 6 strategies, ~1200+ LOC) renames
Ghidra auto-names based on string context, API calls, call graph position,
dataflow, and types. But it has no knowledge of **what the function does**.

Module 2 uses algorithm detection results to name parameters with domain-aware
semantics:

- Newton-Raphson detected -> `param_1` becomes `K_global` (stiffness matrix)
- Gauss quadrature detected -> `param_3` becomes `gauss_weights`
- Call graph: if caller is `assemble_stiffness`, the passed pointer is likely `K_element`
- Type info: `double *` with triple-nested loop access = matrix

@dataclass
class SemanticName:
    """A semantically-recovered parameter or variable name."""
    original_name: str          # e.g., "param_1"
    semantic_name: str          # e.g., "K_global"
    function_name: str          # e.g., "newton_raphson_solve"
    confidence: float           # 0.0 - 1.0
    source: str                 # "algorithm_template" | "call_graph_propagation" | "type_heuristic" | "usage_pattern"
    reason: str                 # Human-readable explanation
    domain: str                 # "structural", "fluid", etc.

@dataclass
class SemanticNamingResult:
    """Full output of semantic parameter naming."""
    success: bool
    names: list[SemanticName]
    total_renamed: int
    by_source: dict[str, int]       # source -> count
    by_domain: dict[str, int]       # domain -> count
    output_files: list[Path]
    naming_map: dict[str, dict[str, str]]   # func_name -> {old_name: new_name}
    errors: list[str]


class SemanticParameterNamer:
    """Algorithm-context-aware parameter and variable naming.

    Works as a post-processing layer after CVariableNamer and NameMerger.
    Only renames parameters that are still Ghidra auto-names (param_N, local_XX)
    or have low-confidence generic names.

    Four naming strategies (in priority order):
    1. Algorithm Template Matching (conf 0.85)
       - Function has algorithm detection -> apply known parameter templates
    2. Call Graph Propagation (conf 0.75)
       - Caller's named arg propagates to callee's param
    3. Type+Usage Heuristic (conf 0.60)
       - double* with triple-nested loop = matrix, int with < comparison = counter
    4. Struct Field Context (conf 0.70)
       - If param is a known struct type from Module 1, name it accordingly
    """

    def __init__(self, config: Config) -> None: ...

    def rename(
        self,
        decompiled_dir: Path,
        functions_json: Path,
        call_graph_json: Path,
        output_dir: Path,
        *,
        algorithm_matches: list[AlgorithmMatch] | None = None,
        domain_report: DomainReport | None = None,
        enriched_structs: list[EnrichedStruct] | None = None,
    ) -> SemanticNamingResult: ...

    # --- Strategy 1: Algorithm Template ---
    def _apply_algorithm_templates(
        self,
        func_name: str,
        params: list[dict],         # from functions_json
        func_code: str,
        algorithms: list[AlgorithmMatch],
    ) -> list[SemanticName]: ...

    # --- Strategy 2: Call Graph Propagation ---
    def _propagate_from_callers(
        self,
        func_name: str,
        params: list[dict],
        call_graph: dict[str, list[str]],
        known_names: dict[str, dict[str, str]],
    ) -> list[SemanticName]: ...

    # --- Strategy 3: Type+Usage Heuristic ---
    def _infer_from_type_and_usage(
        self,
        func_name: str,
        params: list[dict],
        func_code: str,
        domain: str | None,
    ) -> list[SemanticName]: ...

    # --- Strategy 4: Struct Field Context ---
    def _name_from_struct_context(
        self,
        func_name: str,
        params: list[dict],
        enriched_structs: list[EnrichedStruct],
    ) -> list[SemanticName]: ...

    # --- Conflict Resolution ---
    def _resolve_conflicts(
        self,
        candidates: list[SemanticName],
    ) -> list[SemanticName]:
        """When multiple strategies suggest different names for the same param,
        pick the highest-confidence one. Break ties: template > callgraph > struct > heuristic."""

    # --- Code Rewriting ---
    def _apply_names_to_code(
        self,
        c_code: str,
        names: list[SemanticName],
    ) -> str:
        """Replace param_N with semantic names in C code.
        Preserves original as comment on the declaration line."""
```

### 3.10 Key Algorithm Description

1. **Template Application Phase:**
   - For each function with `AlgorithmMatch`, look up `ALGORITHM_PARAM_TEMPLATES`.
   - Match template params to actual params by position and type compatibility.
   - Type compatibility: `double *` matches `long *`, `undefined8 *`, `void *`.
   - Generate `SemanticName` with confidence from match quality.

2. **Call Graph Propagation Phase:**
   - Reverse the call graph to get: callee -> [callers].
   - For each caller with already-named params (from template or c_namer),
     trace which named variable is passed as argument `i` to callee.
   - Propagate the name. Confidence decays by 0.9x per hop.
   - Max propagation depth: 3 hops.

3. **Type+Usage Heuristic Phase:**
   - Parse function body for loop structures and array access patterns.
   - `double *` accessed as `arr[i*n + j]` in triple-nested loop -> "matrix" prefix.
   - `int` used as loop bound (`i < param`) -> "count" or "size" suffix.
   - Domain-aware: if function is in "structural" domain, prefer FEA terminology.

4. **Conflict Resolution:**
   - Sort candidates by confidence descending.
   - Check for name collisions within the same function scope.
   - Append numeric suffix if collision: `K_global` -> `K_global_2`.
   - Never rename if existing name is already semantic (not a Ghidra auto-name).

### 4.1 Purpose

@dataclass
class BlockAnnotation:
    """A single code block annotation."""
    line_number: int            # insertion point (before this line)
    annotation_type: str        # "algorithm_block" | "convergence" | "assembly" | "cross_ref" | "memory_op" | "loop_purpose"
    text: str                   # The comment text to insert
    confidence: float
    source: str                 # which detector produced this

@dataclass
class AnnotatedFunction:
    """A function with its block annotations."""
    function_name: str
    annotations: list[BlockAnnotation]
    algorithm_context: str | None
    domain: str | None

@dataclass
class BlockAnnotationResult:
    """Full output of block annotation."""
    success: bool
    annotated_functions: list[AnnotatedFunction]
    output_files: list[Path]
    total_annotations: int
    by_type: dict[str, int]         # annotation_type -> count
    errors: list[str]


class CodeBlockAnnotator:
    """Engineering-context code block annotator.

    Detects and annotates structural code patterns:
    1. Algorithm blocks (loop structures matching known algorithms)
    2. Convergence checks (norm comparisons, residual tests)
    3. Assembly operations (scatter/gather to global arrays)
    4. Memory operations (allocation, deallocation patterns)
    5. Cross-references (call graph context)
    6. Loop purpose (integration, iteration, element loop, DOF loop)
    """

    def __init__(self, config: Config) -> None: ...

    def annotate(
        self,
        decompiled_dir: Path,
        functions_json: Path,
        call_graph_json: Path,
        output_dir: Path,
        *,
        algorithm_matches: list[AlgorithmMatch] | None = None,
        domain_report: DomainReport | None = None,
        enriched_structs: list[EnrichedStruct] | None = None,
    ) -> BlockAnnotationResult: ...

    # --- Detector 1: Algorithm Blocks ---
    def _detect_algorithm_blocks(
        self,
        lines: list[str],
        func_name: str,
        algorithms: list[AlgorithmMatch],
    ) -> list[BlockAnnotation]:
        """Detect loop structures that correspond to known algorithms.

        Example: Triple-nested loop in a Gauss-Quadrature-detected function
        -> annotate as integration loop with point count.
        """

    # --- Detector 2: Convergence Checks ---
    def _detect_convergence_checks(
        self,
        lines: list[str],
        func_name: str,
    ) -> list[BlockAnnotation]:
        """Detect convergence test patterns.

        Patterns:
        - if (norm < tol) / if (residual < eps)
        - while (err > tol) / do { ... } while (delta > threshold)
        - fabs(x - x_prev) < convergence_criterion
        """

    # --- Detector 3: Assembly Operations ---
    def _detect_assembly_ops(
        self,
        lines: list[str],
        func_name: str,
        domain: str | None,
    ) -> list[BlockAnnotation]:
        """Detect global matrix/vector assembly patterns.

        Patterns:
        - K_global[dof_map[i]] += K_local[j]  (scatter)
        - double sum = 0; for(...) sum += A[i]*x[i]  (dot product)
        - matrix[i*n + j] = ...  (row-major storage)
        """

    # --- Detector 4: Memory Operations ---
    def _detect_memory_ops(
        self,
        lines: list[str],
        func_name: str,
    ) -> list[BlockAnnotation]:
        """Detect memory allocation/deallocation blocks.

        Patterns:
        - malloc/calloc followed by size calculation -> annotate purpose
        - free() calls -> annotate what is being freed
        - memcpy/memset blocks -> annotate data movement purpose
        """

    # --- Detector 5: Cross References ---
    def _generate_cross_refs(
        self,
        func_name: str,
        call_graph: dict[str, list[str]],
        reverse_graph: dict[str, list[str]],
    ) -> list[BlockAnnotation]:
        """Generate cross-reference annotations.

        - "Called by: main -> solve -> this_function"
        - "Calls: malloc, compute_K, solve_system"
        - "Callback chain depth: 3"
        """

    # --- Detector 6: Loop Purpose ---
    def _detect_loop_purpose(
        self,
        lines: list[str],
        func_name: str,
        domain: str | None,
    ) -> list[BlockAnnotation]:
        """Detect loop purpose from iterator patterns and body content.

        - for(e=0; e<num_elements; e++) -> "Element loop"
        - for(i=0; i<ndof; i++) -> "DOF loop"
        - for(gp=0; gp<ngauss; gp++) -> "Integration point loop"
        - while(iter < max_iter) -> "Iteration loop (solver)"
        """

    # --- Code Insertion ---
    def _insert_annotations(
        self,
        lines: list[str],
        annotations: list[BlockAnnotation],
    ) -> str:
        """Insert annotation comments into code at correct positions.
        Sorts by line number descending to avoid offset shifts."""
```

### 4.10 Key Algorithm Description

1. **Per-Function Analysis:**
   - For each function, split code into lines.
   - Run all 6 detectors in sequence.
   - Each detector returns `BlockAnnotation` objects with line numbers.

2. **Algorithm Block Detection:**
   - If function has `AlgorithmMatch`, check loop structure against known patterns.
   - Gauss quadrature: look for nested `for` loops with iteration counts matching
     known quadrature rules (1, 2, 3, 4 points per dimension).
   - Newton-Raphson: look for while/do-while loop containing matrix solve and
     residual computation.
   - Mark the outermost loop of the detected algorithm block.

3. **Convergence Check Detection:**
   - Pattern-match against `CONVERGENCE_PATTERNS` regex list.
   - Extract variable names from capture groups and format into readable comment.
   - Distinguish between norm-based (`||R|| < tol`) and increment-based (`|du| < eps`).

4. **Cross Reference Generation:**
   - Walk call graph to find callers (up to depth 3) and callees.
   - Format as chain: `main -> solve_nonlinear -> newton_raphson -> [this function]`.
   - Also list direct callees with their detected algorithm names if available.

5. **Insertion:**
   - Collect all annotations for a function.
   - Sort by line number descending.
   - Insert from bottom to top so line numbers don't shift.
   - Use consistent formatting: `// === ... ===` for major blocks, `// --- ... ---` for sub-blocks, `// [Cross-ref] ...` for cross-references.

## 9. Risk Assessment

### 9.1 Scope Risk: MEDIUM

Three modules totaling ~2930 LOC. Each is self-contained with clear interfaces.
The templates (ALGORITHM_STRUCT_TEMPLATES, ALGORITHM_PARAM_TEMPLATES) are the
largest data components and can be incrementally expanded.

**Mitigation:** Implement in order: Module 1 -> Module 2 -> Module 3. Each module
is independently testable and deployable. Ship Module 1 alone if time-constrained.

### 9.2 False Positive Risk: LOW-MEDIUM

Algorithm template matching may assign wrong struct/param names if the algorithm
detection itself was wrong (e.g., a loop pattern falsely matched as Gauss quadrature).

**Mitigation:**
- All three modules require `min_confidence` thresholds (configurable).
- Module 1 preserves original as comment: `/* was: *(int*)(param_18 + 4) */`.
- Module 2 preserves original: `int npe = ...; /* was: param_1 */`.
- Module 3 annotations are comments only -- they don't change code semantics.
- Each module's structural-only confidence is capped at 0.50 (matching existing pattern in `analyzer.py` line 49).

### 9.3 Performance Risk: LOW

All three modules are regex-based, no ML inference. Expected overhead per module:
- Module 1: ~0.5s per 1000 functions (regex + call graph walk)
- Module 2: ~0.3s per 1000 functions (template lookup + regex)
- Module 3: ~0.4s per 1000 functions (6 detectors, regex only)

Total v1.1 overhead: ~1.2s per 1000 functions. Negligible compared to Ghidra
decompilation (minutes to hours for large binaries).

### 9.4 Breaking Change Risk: ZERO

- All three modules are opt-in (enabled by default, but configurable).
- All three follow existing `try/except ImportError` pattern.
- No existing interfaces change.
- Version bump from 1.0.0 to 1.1.0 (minor, not major).
