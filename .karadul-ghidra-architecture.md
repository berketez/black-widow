## Core Architecture

### Main Components

#### 1. **GhidraHeadless** (`headless.py` - 1309 lines)

**Primary Wrapper Class**: Handles both PyGhidra and CLI-based analysis

```python
class GhidraHeadless:
    def __init__(self, config: Config) -> None
    def analyze(
        binary_path: Path,
        project_dir: Path,
        project_name: str = "karadul_analysis",
        scripts: list[Path] | None = None,
        timeout: int | None = None,
        output_dir: Path | None = None,
    ) -> dict[str, Any]
```

**Key Methods**:
- `_analyze_pyghidra()` - Native PyGhidra API mode (preferred)
- `_analyze_cli()` - CLI fallback using `analyzeHeadless` (Jython)
- `_check_pyghidra()` - Lazy singleton check for PyGhidra availability
- `_ensure_pyghidra_started()` - JVM launcher with G1GC tuning

**Analysis Pipeline** (PyGhidra mode):
1. Extract functions (`_extract_functions`)
2. Extract strings (`_extract_strings`)
3. Extract call graph (`_extract_call_graph`)
4. Decompile functions with batch processing (`_decompile_functions`)
5. Extract types/structs/enums (`_extract_types`)
6. Extract cross-references (`_extract_xrefs`)
7. Combine all results (`_get_program_info`)

## Analysis Scripts & Data Flow

### PyGhidra Mode Flow Diagram

```
analyze() [Binary Path]
    ↓
[PyGhidra Available?]
    ├─ YES → _analyze_pyghidra()
    │         ├─ _extract_functions() → functions.json
    │         ├─ _extract_strings() → strings.json
    │         ├─ _extract_call_graph() → call_graph.json
    │         ├─ _decompile_functions() → decompiled/ + decompiled.json
    │         ├─ _extract_types() → types.json
    │         ├─ _extract_xrefs() → xrefs.json
    │         └─ _get_program_info() + combined_results.json
    │
    └─ NO → _analyze_cli()
            └─ analyzeHeadless + scripts/*.py
                ├─ function_lister.py → functions.json
                ├─ string_extractor.py → strings.json
                ├─ call_graph.py → call_graph.json
                ├─ decompile_all.py → decompiled/ + decompiled.json
                ├─ type_recovery.py → types.json
                ├─ xref_analysis.py → xrefs.json
                └─ export_results.py → combined_results.json
```

### CLI Scripts (Jython 2.7 - Ghidra JVM)

Located: `/karadul/ghidra/scripts/`

#### 1. **function_lister.py** (76 lines)
Extracts all functions with metadata
- FunctionManager.getFunctions()
- Each function: name, address, size, param_count, return_type, is_thunk, calling_convention, is_external
- Output: `functions.json` with total count

#### 2. **string_extractor.py** (147 lines)
Extracts all defined strings
- DefinedDataIterator.definedStrings()
- Categories: URLs, paths, API keys, error messages, format strings, other
- Xref counting per string (which functions use it)
- Output: `strings.json` with category statistics

**Data Type Checks**:
```jython
if "string" in dt_name or "cstring" in dt_name:
    # Extract with xref context
```

#### 3. **call_graph.py** (169 lines)
Extracts complete call graph
- nodes: {address: {name, address, caller_count, callee_count, callers[], callees[]}}
- edges: {from, to, from_name, to_name}
- Identifies root functions (no callers, has callees)
- Identifies leaf functions (no callees, has callers)
- Output: `call_graph.json` with graph statistics

#### 4. **xref_analysis.py** (415 lines)
Most complex analysis – three-part xref system
- **Function xrefs**: strings_used, globals_accessed, functions_called, called_by
- **String xrefs**: referenced_by (functions, with addresses), defined_in_function
- **Global xrefs**: readers/writers classification, is_external flag
- Output: `xrefs.json` with statistics (most referenced strings, most called functions, most written globals)

#### 5. **decompile_all.py** (415 lines)
Decompiles all functions with enrichment
- DecompInterface with 30s per-function timeout
- Extracts: C code, disassembly, xrefs, stack frame
- Per-function output files: `decompiled/{func_name}.c`
- Respects MAX_FUNCTIONS and MAX_TIME_SECONDS env vars
- Output: `decompiled.json` with success/failed/skipped counts

**Stack Frame Extraction**:
```jython
frame.getStackVariables() → {offset, size, type, kind (parameter|local), comment}
```

#### 6. **type_recovery.py** (242 lines)
Extracts type information
- Structure (composite): fields with offset, size, type, comment
- Union: same as Structure
- Enum: named values with integers
- TypeDef: base_type mapping
- FunctionDefinition: return_type, calling_convention, parameters
- Output: `types.json` with category summary

**Type Enumeration**:
```jython
from ghidra.program.model.data import Structure, Union, Enum, TypeDef, FunctionDefinition, Pointer
```

#### 7. **export_results.py** (162 lines)
Combines all analysis results
- Loads all *.json files from output directory
- Compiles program_info (name, language, processor, endian, memory_blocks)
- Aggregates summary statistics
- Samples first 100 functions/strings (full data in separate files)
- Output: `combined_results.json` (dashboard-friendly)

### PyGhidra Direct Extraction Methods

Same functionality as CLI scripts but in Python 3 (within JVM context):

- `_extract_functions(program)` → List[FunctionEntry]
- `_extract_strings(program)` → List[StringEntry] with xref context
- `_extract_call_graph(program)` → {nodes, edges}
- `_extract_disassembly(program, func)` → List[Instruction]
- `_extract_func_xrefs(program, func)` → {callers, callees}
- `_extract_stack_frame(func)` → {variables, frame_size, ...}
- `_decompile_functions(program, output_dir, timeout, batch_size)` → Large batch handler
- `_extract_types(program)` → {structures, enums, typedefs}
- `_extract_xrefs(program)` → Function/String/Global xref maps
- `_get_program_info(program)` → {language, processor, memory_blocks, ...}

## Performance Considerations

### Memory & Timeout Tuning

**Ghidra Heap Calculation**:
```python
# config.py
TOTAL_MEMORY_MB = _detect_available_memory_mb()
GHIDRA_HEAP_MB = max(4096, min(32768, int(TOTAL_MEMORY_MB * 0.4)))

# On Apple Silicon M3 Max (36GB):
# GHIDRA_HEAP_MB = 14336 MB (40% of 36GB, capped at 32GB)
```

**Large Binary Handling**:
- Threshold: 100MB
- Timeout multiplier: 4.0x
- Batch size: 5000 functions per batch
- GC trigger: After each batch (Python gc.collect())

**Decompilation Bottleneck**:
- Per-function timeout: 30 seconds
- Batch timeouts prevent JVM heap exhaustion
- Progress logging: Every 5000 functions

### Recommended Ghidra VM Args

Current (headless.py:56-65):
```
-XX:+UseG1GC                    # Garbage-first collector
-XX:ParallelGCThreads=12        # GC threads = P-core count
-XX:ConcGCThreads=6             # Concurrent GC threads
-XX:+UseCompressedOops          # 64-bit object pointers → 32-bit references
-XX:ActiveProcessorCount=12     # Explicit processor affinity
-Xmx14336m                       # Heap (auto-detected)
```

**For BSim/P-Code features**:
- Add: `-XX:+PrintGCDetails` for monitoring
- Add: `-XX:InitiatingHeapOccupancyPercent=35` for earlier GC triggers
- Consider: `-XX:G1NewGenSize=512m` for large function sets
